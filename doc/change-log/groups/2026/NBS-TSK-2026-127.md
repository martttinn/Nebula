# Resumen

Grupo que recoge la consolidación de la familia `hero` en una carpeta propia y la restauración de wrappers finos en rutas antiguas para mantener compatibilidad de imports y tabs abiertos del editor.

## Objetivo del grupo

Ordenar la sección hero como un boundary coherente sin romper la API pública ya consumida por la home ni generar drift operativo en rutas heredadas.

## Cambios incluidos

- creación de `components/home/hero/` como carpeta canónica del hero
- agrupación de shell, lead, partículas y grid scan bajo el mismo boundary
- restauración de wrappers de compatibilidad para rutas antiguas del hero y de las otras secciones ya modularizadas
- actualización de la referencia técnica para reflejar el nuevo entrypoint canónico

## Impacto en arquitectura / producto

Cambio interno de frontend. No altera comportamiento visual del hero ni el contrato de consumo desde `app/page.tsx`.

## Validacion agregada

- `npm run lint`
- `npm run typecheck`
- `npm run build`

## Riesgos o arrastres

- la validación visual final del hero sigue siendo manual
- los wrappers de compatibilidad mantienen paths antiguos disponibles, pero el boundary canónico ya es la carpeta `hero/`
- `npm run changes:sync` no puede ejecutarse porque los scripts `changes:*` no existen hoy en `package.json`

## Notas para presupuesto

Refactorización estructural final de la home con doble retorno: mejor organización interna del hero y reducción de fricción práctica en el editor al preservar rutas antiguas como fachada.
