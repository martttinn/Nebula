# Resumen

Grupo que recoge el segundo pase conservador de performance tras el analisis PageSpeed mobile compartido el `2026-05-24`.

## Objetivo del grupo

Reducir coste de JavaScript, main thread y reflows forzados sin modificar el resultado visual de la home publica.

## Cambios incluidos

- carga imperativa por idle de `Lenis` desde `components/layout/lenis-provider.tsx`, sin reenvolver ni remontar la home
- carga por idle de Vercel Analytics y Speed Insights desde `components/layout/vercel-observability.tsx`
- `Preloader` desacoplado de `lenis/react`, manteniendo scroll lock propio y una sola ejecucion por carga inicial
- indicador global de scroll con escritura imperativa de `transform`, sin renders React por frame
- cacheo de metricas de scroll en `services-carousel`, `projects-showcase` y `testimonials`
- desactivacion en mobile de listeners/calculos desktop ocultos en servicios y proyectos
- actualizacion de referencia tecnica, decisions log y change-log

## Impacto en arquitectura / producto

Impacto frontend runtime y documentacion. No cambia copy, SEO, layout, identidad visual, rutas, backend, auth ni datos. La experiencia visible queda equivalente, pero parte del trabajo no critico sale del arranque inmediato o deja de ejecutarse en viewports donde no aporta valor.

## Validacion agregada

- `npm run lint`
- `npm run typecheck`
- `npm run build`
- inspeccion de `.next/server/app/page_client-reference-manifest.js`
- inspeccion de chunks `.next/static/chunks`

## Riesgos o arrastres

La mejora real de PageSpeed solo queda confirmada tras desplegar y repetir medicion sobre `somosnebula.com`. El split completo de secciones below-the-fold queda fuera de este pase porque requeriria fallbacks estaticos equivalentes por seccion para no introducir drift visual durante carga.

## Notas para presupuesto

Trabajo de performance conservador: optimiza coste de runtime movil y main thread sin redisenar ni retirar piezas de marca.
