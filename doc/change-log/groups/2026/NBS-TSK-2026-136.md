# Resumen

Grupo que recoge la instalación de `tech-stack-icons` y la definición del patrón repo-safe para consumirlo sin penalizar el bundle crítico del frontend.

## Objetivo del grupo

Habilitar iconografía de stack tecnológico en el proyecto sin introducir imports directos de un paquete monolítico dentro de Client Components críticos ni dejar una convención ambigua para usos futuros.

## Cambios incluidos

- instalación de `tech-stack-icons` en `package.json` y `package-lock.json`
- creación de `components/ui/tech-stack-icon.tsx` como wrapper server-first
- creación de `components/ui/lazy-tech-stack-icon.tsx` como fallback cliente con carga diferida por viewport
- sustitución del texto plano de `stack principal` por badges con icono + label dentro de `components/home/projects-showcase/index.tsx`
- actualización de la referencia técnica y de la foto de stack en `.agents/rules/01-project-context.md`
- registro de la entrega en `doc/change-log/`

## Impacto en arquitectura / producto

El repo gana una primitive reusable para iconos tecnológicos y deja fijada una frontera clara entre consumo server-side y consumo cliente perezoso. El cambio ya altera una surface pública visible en `projects-showcase` y reduce el riesgo de que futuras secciones arrastren un bundle innecesario por un uso ingenuo de la librería.

## Validacion agregada

- `npm run lint`
- `npm run typecheck`
- `npm run build`

## Riesgos o arrastres

- el paquete sigue siendo pesado si se usa en cliente, aunque ahora queda encapsulado detrás de `LazyTechStackIcon`
- una externalización inicial por `next.config.mjs` resultó incompatible con la build Turbopack del repo y se descartó del estado final
- la instalación ha reexpuesto dos advisories moderados ya existentes en `next` y `postcss`; no pertenecen a `tech-stack-icons` y quedan fuera de este alcance
- los scripts `changes:*` siguen sin estar disponibles en `package.json`, así que el índice derivado del `change-log` no puede resincronizarse por script en esta tarea

## Notas para presupuesto

Trabajo principalmente estructural y preventivo: combina investigación técnica, integración de dependencia, configuración del framework y hardening de una primitive reusable para evitar regresiones de rendimiento futuras.
