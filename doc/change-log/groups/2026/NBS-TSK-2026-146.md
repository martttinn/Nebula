# Resumen

Grupo que recoge la normalización de IDs duplicados en el histórico del `change-log` y la reparación de sus agrupaciones semánticas.

## Objetivo del grupo

Eliminar warnings estructurales del validador, devolver unicidad a `change_id` y separar entradas que habían quedado mezcladas en grupos incorrectos por colisiones antiguas.

## Cambios incluidos

- localización y análisis de las seis colisiones históricas reportadas por `changes:validate`
- creación de nuevos grupos canónicos para las entradas que estaban mezcladas semánticamente con otros cambios
- reasignación de `change_id` y `group_id` a seis entradas históricas sin alterar su fecha ni su contenido funcional
- regeneración del índice derivado del `change-log` y validación completa del árbol ya saneado

## Impacto en arquitectura / producto

Impacto interno en documentación y tooling repo-local. El runtime público no cambia, pero el subsistema de `change-log` pasa a quedar limpio de duplicados y coherente con la historia real del repositorio.

## Validacion agregada

- `npm run changes:validate`
- `npm run changes:index`
- `npm run changes:sync`
- `npm run lint`
- `npm run typecheck`
- `npm run build`

## Riesgos o arrastres

- la normalización corrige IDs y grupos, pero no reescribe el estilo legacy de algunas entradas antiguas; esa heterogeneidad histórica sigue siendo tolerada de forma explícita por el validador
- cualquier referencia externa manual a los IDs antiguos normalizados quedaría obsoleta, aunque dentro del repo no quedaban referencias activas fuera del propio `change-log`

## Notas para presupuesto

Entrega interna de hardening documental y DX: no añade funcionalidad pública, pero elimina deuda operativa y deja el histórico verificable sin warnings.
